아래 순서로 코드 리뷰를 진행해줘.

## 준비
1. `git diff HEAD` 로 변경된 코드 전체 확인
2. 변경된 파일 목록 파악

## 리뷰 항목

### 1) 정확성 / 버그 위험
- 로직 오류, 엣지 케이스 누락, 예외 처리 미흡
- 비동기 처리 문제 (await 누락, race condition 등)

### 2) 클린 코드 / 유지보수성
- 가독성, 함수/변수 네이밍
- 단일 책임 원칙 위반 여부
- 중복 코드, 불필요한 복잡도
- **테스트 포인트**: 이 변경에서 단위 테스트로 검증해야 할 케이스를 구체적으로 제안

### 3) 프로젝트 컨벤션 준수
- CLAUDE.md 규칙 준수 여부 (타입, API 레이어, 스타일 토큰 등)
- any 사용 여부
- 컴포넌트/composable/store 레이어 구분

### 4) 성능
- 불필요한 리렌더링, 반복 계산
- computed 활용 여부

## 출력 형식

각 항목에서 발견된 사항만 출력한다 (문제 없으면 해당 항목 생략).
심각도를 표시한다: 🔴 차단 / 🟡 권장 / 🔵 참고

마지막에 **테스트 포인트** 섹션을 항상 출력한다.
두 가지 관점으로 나눠서 작성한다:

1. **구현 관점**: 내부 로직, API 호출 조건, 상태 변화 등
2. **사용처/사용자 관점**: 이 코드를 호출하는 쪽에서 관찰할 수 있는 동작 결과 (실제 사용처 파일을 검색해서 파악)

```
## 테스트 포인트

### 구현 관점
- [ ] 검증이 필요한 케이스 1

### 사용처/사용자 관점
- [ ] 사용자가 경험하는 동작 결과 1
```
